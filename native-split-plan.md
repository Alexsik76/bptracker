# BP Tracker — план розділення: нативний Android + веб
 
Статус: концепт для обговорення. Не план робіт, не промпти агентам.
 
## Мотивація (стисло, з фактів)
 
Виміряно експериментально (червень 2026): Web Push через FCM на заблокованому
Samsung S25 втрачає до 80% повідомлень при щільній відправці (4 з 20 за 10 хв),
`Urgency: high` не допомагає. Це задокументована поведінка платформи
(App Standby Buckets, FCM high-priority quota, 7-денна деприоритизація), а не
баг нашого коду. PWA не має API для локальних запланованих нагадувань
(Notification Triggers закинуто, Periodic Background Sync ~раз/добу).
 
Висновок: нагадування мають перестати бути мережевою подією. Це вміє лише
нативний застосунок (AlarmManager). Решта телефонних сценаріїв (камера, BLE
у перспективі, Health Connect) — той самий клас обмежень PWA.

| Простий дашборд | ✅ останній замір, зона, мотиваційні метрики (див. нижче) | ✅ повний |
| Гейміфікація | ✅ стрік днів нормального тиску, дисципліна прийому | — |
| Експорт CSV на email | ✅ кнопка → існуючий `POST /export/csv` | ✅ |

| Введення/редагування схем | — | ✅ |

| Налаштування | мінімум (токен, сервер) | ✅ повні |
| Офлайн-черга замірів | ✅ Room | ❌ (знімається) |
 
Екран нативу — не аналітика, а «сьогодні»: останній замір + незакриті
прийоми + мотиваційні лічильники (стрік днів у нормі, серія днів без
пропуску ліків). Повна історія — лише у вебі.
 
**Веб деградує з PWA до звичайного SPA**: демонтуються Service Worker,
офлайн-черга, Web Share Target, manifest, Web Push (підписки, VAPID).
Це мінус ціла категорія багів (navigation fallback, версії кешу, життєвий
цикл SW) і мінус уся FCM-залежність.
 
**Бекенд**: ReminderWorker і PushService — під знос (на фазі 4). Розклад
нагадувань лишається в БД як джерело істини; натив синхронізує його і
ставить локальні алярми сам. Новий механізм auth для нативу: довгоживучий
device token (один новий ендпоінт). Журнал прийомів (`MedicationIntakes`)
— на бекенді.

## OCR: навчені моделі в Kotlin
 
Розпізнавання — повністю локальне на телефоні, фото не покидає пристрій
(як і зараз у браузерному OCR). Навчання моделей — як було, на ПК у
`bp-ocr-cnn`; телефон лише виконує інференс готових файлів.
 
**Що вже є (bp-ocr-cnn):** двостадійний YOLOv8 nano, експортований в ONNX
int8 — display_detector 3.2 MB + digit_detector 3.1 MB = **6.3 MB**, 100%
точності на еталоні. Ці самі файли вже працюють у браузері через
onnxruntime-web — формат перевірений.
 
**Шлях у натив: ONNX Runtime Android** (`com.microsoft.onnxruntime:onnxruntime-android`).
Ті самі `_int8.onnx` файли без жодної конвертації:
 
```
Фото (CameraX) → letterbox/нормалізація → YOLO#1 (display) → crop 400×480
→ YOLO#2 (digits) → class-agnostic NMS → K-means k=3 (рядки) → {sys, dia, pul}
```
 
Модель вміє лише «тензор → рамки». Обв'язка навколо неї — звичайний код,
що сьогодні існує у двох еталонних копіях: Python-скрипти `bp-ocr-cnn`
(`recognize_digits.py`, `validate_pipeline.py`) і TypeScript у фронті
(`useLocalOcr`). Для нативу пишеться третя копія цієї логіки на Kotlin
(~200–300 рядків): препроцесинг, декодування виходу YOLO,
class-agnostic NMS, K-means по Y, збірка SYS/DIA/PUL з валідацією
діапазонів (40–300 / 20–200 / 30–250).
 
Критерій готовності: Kotlin-пайплайн дає ідентичний результат з
`validate_pipeline.py --backend int8` на еталонних 42 фото.
 
Моделі — в `assets/` APK (+6.3 MB). Оновлення моделі після донавчання =
новий реліз APK; завантаження моделей по мережі — не для MVP.
 
Шляхи фото в нативі: основний — локальний ONNX; Gemini fallback через
існуючий `/measurements/analyze` — лишити як страховку; збір датасету
(fire-and-forget у aivm-photo-api) — лишити, він живить донавчання.

   на бекенді — знос ReminderWorker, PushService, push-підписок.
5. **Нові функції за потребою**: BLE до тонометра, Health Connect,
   оновлення моделей по мережі.
## Зроблено (фаза 1, триває)
 
- **Per-user ізоляція лікувального ланцюга** (червень 2026): `UserId` додано
  до `TreatmentSchema`, `ReminderTemplate`, `IntakeReport`; усі запити в
  `SchemaService`/`ReminderService` фільтрують по користувачу; інваріант
  «одна активна схема/шаблон» став per-user і атомарним. Міграція
  `AddUserOwnership` застосована на проді (триступенева: nullable → backfill
  → NOT NULL+FK), дані збережено. Unique-constraint
  `IntakeReport (UserId, TemplateId, Period, Date)` + ідемпотентний
  `ConfirmAsync` із захистом від гонки.
- **Міграція часових поясів (Timezone / `DateTime` → `DateTimeOffset`)**:
  усі колонки дат у базі мігровано на `timestamptz`, у C# моделях використано
  `DateTimeOffset`. `ConfirmAsync` тепер рахує `today` з урахуванням таймзони клієнта
  (IANA), що усуває проблему запису вчорашнім днем після півночі.
- **`EmailOutbox.UserId`**: додано колонку `UserId` (nullable) та налаштовано
  її заповнення при CSV-експорті (user-triggered листи), тоді як системні
  (magic-link) лишаються null.

## Відкриті питання
 
- Параметри нагадувань (часи періодів, повтори) — вирішується на фазі 2,
  коли алярми запрацюють; повтори локальні й безкоштовні, без FCM-квот.
- Безпека device token — потребує окремого дизайну (формат, зберігання,
  відкликання, TTL; passkeys лишаються веб-каналом). Підготувати пропозицію
  до старту фази 2.
